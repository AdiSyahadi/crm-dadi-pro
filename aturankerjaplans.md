# ANALYSIS PRINCIPLES

You MUST:

- Understand full system context first
- Identify components before judging
- Separate features from implementation details
- Separate core features vs supporting features
- Separate technical capability vs product offering
- Think in system layers
- Consider business implications
- Consider UX implications

Never list features randomly.
Always structure findings.

---

# SYSTEM LAYER THINKING MODEL

All analysis must classify system into layers:

1. Core Business Features  
2. Operational Features  
3. Automation Features  
4. Reporting & Analytics  
5. Integration & API  
6. Admin & Control  
7. Scalability & Limits  
8. Infrastructure Cost Drivers  

If a feature does not clearly belong to a layer → clarify before continuing.

---

# FEATURE CLASSIFICATION RULE

When analyzing a project:

You MUST:

- List all detected features
- Categorize each feature
- Identify feature dependencies
- Identify which features are:
  - Revenue drivers
  - Retention drivers
  - Cost drivers
  - Complexity drivers

Never mix feature discovery with feature redesign.

Discovery first.
Evaluation second.
Strategy third.

---

# SAAS PLAN DESIGN FRAMEWORK

When structuring SaaS plans:

You MUST evaluate:

1. Value Differentiation  
   - What creates real perceived value?
   - What feels premium?

2. Cost Impact  
   - What increases server load?
   - What increases API usage?
   - What increases storage cost?

3. Upgrade Motivation  
   - What makes users want to upgrade?
   - What creates natural growth path?

4. Operational Risk  
   - Abuse potential?
   - Scalability pressure?
   - Support burden?

Plans must reflect:

- Value hierarchy
- Infrastructure cost
- Logical upgrade path
- User maturity progression

Never split plans randomly.
